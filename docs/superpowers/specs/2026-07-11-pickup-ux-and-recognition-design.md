# 取件助手体验与识别改进设计

**日期**: 2026-07-11  
**版本目标**: v1.4.0（同一版本内分两层交付）  
**状态**: 待用户确认后进入实现计划  

## 1. 背景与目标

在快递 logo 接入与本地提取链路稳定后，本版聚焦：

1. **去掉图片 OCR**（用户明确不再需要）
2. **删除防重与确认取件对齐**（软状态 + 3 天过期，期间不重复入库）
3. **收紧取件短信判定**，减少「在途 / 运单提醒」误入库
4. **列表交互与界面**（搜索、筛选、批量、复制、卡片与空状态）

原则：隐私与本地优先不变；识别宁可漏模糊在途短信，也不把非取件通知收成待取包裹。

## 2. 交付分层（路径 B）

| 层 | 内容 | 验收重点 |
|----|------|----------|
| **基础层** | 去 OCR；`status` 生命周期；删除/取件 3 天墓碑；门控与取件码规则 | 正确性、回归测试短信 |
| **体验层** | 搜索 / 筛选 / 批量 / 复制；卡片与空状态；文案与引导 | 可用性、视觉 |

两层可同一分支连续实现，但测试与合并叙事按层分开，便于回滚体验层而不丢正确性修复。

## 3. 去掉图片 OCR

### 3.1 行为

- 新增入口仅保留：**粘贴短信文本**（+ Android 短信批量 / 自动导入，既有能力）。
- 删除图片上传 Tab、预览、OCR 提交流程。

### 3.2 代码与依赖

| 动作 | 目标 |
|------|------|
| 删除 | `services/ocrService.ts` |
| 删除/收缩 | `extractInfosFromImage`、`extractInfoFromImage` |
| 精简 | `ImportMethod`（去掉 `IMAGE` 或整个枚举若仅剩文本） |
| 简化 | `components/AddPackageModal.tsx` 为纯文本导入 |
| 卸载 | `tesseract.js` |
| 文案 | `Onboarding`、`README`、相关 docs：去掉「截图识别」 |

### 3.3 验收

- UI 无图片导入入口。
- `package.json` 无 tesseract；生产包不再打入 OCR worker / 语言包。
- 文本导入与短信导入仍可用。

## 4. 包裹生命周期：删除 = 取件的防重逻辑

### 4.1 数据模型

推荐显式状态（迁移友好）：

```ts
type PackageStatus = 'active' | 'picked' | 'deleted';

interface PackageData {
  // ...既有字段
  status: PackageStatus;
  /** @deprecated 兼容读：status === 'picked' 时为 true */
  isPickedUp?: boolean;
  pickedUpAt?: number;
  deletedAt?: number;
  expiresAt?: number; // picked / deleted 共用：操作时刻 + 3 天
  createdAt: number;
}
```

常量：`RETENTION_MS = 3 * 24 * 60 * 60 * 1000`（与现 `PICKED_UP_EXPIRES_IN_MS` 统一命名）。

### 4.2 迁移

加载时：

- 无 `status` 且 `isPickedUp === true` → `status = 'picked'`，补全 `pickedUpAt` / `expiresAt`（沿用现逻辑）。
- 无 `status` 且未取件 → `status = 'active'`。
- 过滤：`expiresAt <= now` 且 `status ∈ {picked, deleted}` → 从存储移除。

### 4.3 操作语义

| 操作 | status | 列表 | 导入同码（normalize 后） | 3 天后 |
|------|--------|------|--------------------------|--------|
| 新增成功 | `active` | 待取 | — | — |
| 确认取件 | `picked` | 历史 | 不新建；可升级地点/地址/原文/courier | purge |
| 取消取件 | `active` | 待取 | — | 清 `pickedUpAt`/`expiresAt` |
| 删除 | `deleted` | 不展示 | 不新建 | purge |
| 删除恢复 | 不做 | — | 过期后可再导入 | — |

`handleAddPackage`：

1. `normalizePickupCode` 得到 key。
2. 在**未过期**记录中查找同 key（含 `active` / `picked` / `deleted`）。
3. 命中 → 不新建；仅对 `active`/`picked` 做既有字段升级；对 `deleted` **不升级、不复活**。
4. 未命中 → 新建 `active`。

UI 过滤：

- 待取：`status === 'active'`
- 历史：`status === 'picked'`（且未过期）
- `deleted` 永不进入列表 UI

### 4.4 存储

仍用 `localStorage` 单 key `pickmeup_packages_v1`；墓碑记录与正常包裹同表，靠 `status` 区分。不另建永久黑名单表。

## 5. 识别门控与误识别抑制

### 5.1 问题

当前 `isLikelyPickupMessage` 在出现「快递/包裹」等词时过宽，规则可能把**运单号**或在途提醒中的数字段当成取件码。用户场景：仅提醒「某单号快到了」却被入库。

### 5.2 策略

**正向信号（至少其一，且更具体）**

- 取件码语境：`取件码|取货码|提货码|凭…(取|提取|到店|到柜)`
- 可取状态：`已入柜|已到站|待取件|已存入|可凭|到店…提取|到柜…取`

**负向信号（无明确取件码语境时拒绝）**

- 轨迹/在途：`即将送达|正在派送|快到了|派件中|运输中|已发货|已揽收|揽收成功|运单号|快递单号|物流更新`

**取件码候选约束**

- 保留柜格风格：`1-5-0366`、`5-1-1-6154` 等（段式 + 末段数字）。
- 拒绝：无「取件码/凭」语境下的 **10–15 位纯数字运单** 等长串。
- 模板匹配仍优先；规则兜底须满足门控。

### 5.3 驿小哥 / 到店提取类短信

用户提供的真实取件短信作为**正向验收样例**（必须识别成功）：

| # | 原文 | 期望 pickupCode | 期望 location（合理即可） | 期望 courier |
|---|------|-----------------|---------------------------|--------------|
| T1 | 【驿小哥】您好，您的顺丰快递0366已到大连理工大学(开发区校区)，可凭1-5-0366到店SF自营大连理工大学开发区店提取。 | `1-5-0366` | 含「大连理工大学」或「SF自营…开发区店」等可取地点 | 顺丰 |
| T2 | 【驿小哥】您好，您的极兔快递6154已到大连理工大学(开发区校区)，可凭5-1-1-6154到店SF自营大连理工大学开发区店提取。 | `5-1-1-6154` | 同上 | 极兔 |

说明：

- 正文含「顺丰快递0366」时，**0366 不是取件码**；取件码以「可凭…」后为准。
- 发送者「驿小哥」可增加模板或规则加权，但不强制依赖发送者字段（粘贴导入无 sender）。
- 地点优先「到店…提取」前的门店名，或「已到…」后的校区/站点名；允许实现时二选一，但不得为「未知」若文本中有明确到店信息。

### 5.4 负向样例（实现后自测，可扩充）

- 「您的快递单号 SF1234567890 即将送达 / 快到了」→ 不入库。
- 「运单号 xxx 运输中」→ 不入库。
- 无「凭/取件码」且仅有长数字 → 不入库。

### 5.5 模板库补充（基础层建议）

在 `courierTemplates` 或通用规则中覆盖「可凭 {code} 到店 … 提取」模式，避免仅靠过宽正则误抓「快递0366」中的尾号。

## 6. 交互增强（体验层）

| 能力 | 说明 |
|------|------|
| 搜索 | 取件码、地点、地址、courier、originalText 子串（规范化码可同时匹配） |
| 筛选 | 按当前列表出现过的快递公司；可选「地点未知」 |
| 批量 | 多选 → 批量已取 / 批量删除（均走 status + 3 天） |
| 复制 | 点击取件码或按钮复制，轻提示「已复制」 |

范围：仅 `active` / `picked` 列表；`deleted` 不可见故不参与。

## 7. 界面改进（体验层）

- 卡片：logo + 快递名固定顶栏；取件码最大字号；地点次之；时间/过期提示弱化。
- 空状态：待取空 / 历史空 / 搜索无结果 三种文案。
- 新增弹窗：纯文本 + 主按钮「识别并添加」。
- 引导页：智能识别改为「短信与粘贴文本」，去掉截图表述。

## 8. 非目标（本版不做）

- 云同步、账号、服务端
- 系统通知 / 桌面小组件
- 删除后「从回收站恢复」UI
- 重新引入任何 OCR / 云端视觉识别

## 10. 验收清单

### 基础层

- [ ] 无图片 OCR 入口与依赖
- [ ] 删除后 3 天内同码短信/粘贴不重新出现；存储中可为 `deleted` 墓碑
- [ ] 取件后同码不重复新建；历史可见 3 天
- [ ] 过期 `picked`/`deleted` 被 purge
- [ ] **T1、T2 测试短信**提取结果符合 §5.3
- [ ] 典型在途/运单提醒不入库

### 体验层

- [ ] 搜索、筛选、批量已取/删除、复制可用
- [ ] 卡片层级与空状态可读
- [ ] 引导与 README 文案一致

## 11. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 门控过严漏真短信 | 以 T1/T2 与模板「可凭…提取」为回归金样例 |
| `status` 迁移破坏旧数据 | 读时兼容 `isPickedUp`；写回规范化 |
| 批量删除误触 | 沿用确认对话框；批量二次确认 |
